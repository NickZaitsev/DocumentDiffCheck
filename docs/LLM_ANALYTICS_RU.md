# Как устроена LLM-аналитика в DocumentDiffCheck

Документ описывает, что именно отправляется в языковую модель, как формируются
запросы, как (НЕ) происходит деление под контекстное окно и какие используются
промпты. Описание соответствует текущему коду на момент написания — включая
честное указание мест, где поведение отличается от того, что заявлено в README.

---

## 1. Общая схема

Аналитика вызывается в двух сценариях:

1. **Сравнение двух документов** (`analyze_comparison`) — на вход идёт уже
   посчитанный структурный diff между старой и новой версией.
2. **Аудит одного документа** (`analyze_document`) — на вход идут блоки одного
   документа, цель — найти финансовые риски.

Цепочка провайдеров (`src/api/app.py`, `ResilientInsightProvider`):

```
Gemini  ->  OpenRouter  ->  детерминированный fallback (без LLM)
```

Провайдеры пробуются по очереди. Если провайдер бросает `AIProcessingError`,
берётся следующий. Если все LLM-провайдеры недоступны или не настроены ключи,
срабатывает `FallbackInsightProvider` — он считает всё на эвристиках (ключевые
слова + регулярки на деньги/проценты), без обращения к модели.

Ключевые файлы:

| Слой | Файл | Роль |
|------|------|------|
| Сборка payload | `src/infrastructure/insights.py` | Готовит данные для промпта, обрезает их |
| Промпты | `src/config.py` | Тексты системных инструкций |
| Gemini | `src/integrations/gemini_provider.py` | Вызов через `gemini_gateway` |
| OpenRouter | `src/integrations/openrouter_provider.py` | Вызов через REST `chat/completions` |
| Схема ответа | `src/schemas/insights.py` | `ChangeReport` / `ChangeItem` |

## 2. Что отправляется в модель

### 2.1 Сценарий «сравнение» (`build_prompt_payload`)

В модель уходит JSON `ComparisonPromptPayload`:

```jsonc
{
  "comparison_id": "...",
  "old_filename": "dogovor_v1.docx",
  "new_filename": "dogovor_v2.docx",
  "stats": { "added": 3, "removed": 1, "modified": 5,
             "unchanged": 40, "similarity_score": 0.82 },
  "changes": [
    {
      "change_id": "c12",          // стабильный id, по нему модель ссылается
      "change_type": "modified",    // added | removed | modified
      "old_index": 11,
      "new_index": 11,
      "old_text": "...текст старой редакции...",
      "new_text": "...текст новой редакции...",
      "similarity": 0.74
    }
    // ...
  ]
}
```

Важно: в payload **не попадают неизменённые блоки** (`unchanged` отфильтрованы),
модель видит только diff. Каждое изменение имеет стабильный `change_id` вида
`c0`, `c1`, … (присваивается в `domain/comparison.py`), чтобы модель могла
сослаться на источник, а UI — подсветить нужную строку.

### 2.2 Сценарий «аудит одного документа» (`build_document_review_payload`)

В модель уходит JSON `DocumentReviewPromptPayload`:

```jsonc
{
  "document_id": "...",
  "filename": "dogovor.docx",
  "blocks_count": 137,
  "blocks": [
    { "block_id": "p-0", "index": 0, "kind": "paragraph", "text": "..." },
    { "block_id": "t-5", "index": 5, "kind": "table_row", "text": "a | b | c" }
    // ...
  ]
}
```

Здесь модель получает блоки документа (абзацы и строки таблиц), а её задача —
выбрать рискованные пункты и сослаться на них по `block_id`.

## 3. Как происходит деление под контекстное окно

> **Важное и честное уточнение.** Сейчас в коде НЕТ настоящего чанкинга под
> размер контекстного окна. Несмотря на то что в `README.md` и `AGENTS.md`
> упоминаются «chunking strategies», фактически выполняется **обрезка
> (truncation)**, а не деление на несколько запросов.

Что происходит на самом деле (`src/infrastructure/insights.py`):

| Что ограничивается | Параметр | Значение по умолчанию | Поведение при превышении |
|--------------------|----------|-----------------------|--------------------------|
| Число изменений в сравнении | `max_changes` | **60** | всё, что после 60-го изменения, **отбрасывается** |
| Число блоков в аудите | `max_blocks` | **120** | всё, что после 120-го блока, **отбрасывается** |
| Длина текста одного блока/пункта | `_trim(limit=...)` | **1400 символов** | хвост заменяется на `...` |

То есть:

- На **один** документ/сравнение делается **ровно один** запрос в LLM.
- Токены **не считаются** — нет ни `tiktoken`, ни оценки длины. Лимиты заданы
  «в штуках» (60/120) и в символах (1400), а не в токенах.
- Большой договор обрезается «по верхам»: риск, лежащий в блоке № 121 или в
  обрезанном хвосте длинного пункта, в модель **не попадёт вообще**.

Косвенно размер ответа ограничивается `max_output_tokens=4000` (Gemini,
`gemini_provider.py:66`) и `OPENROUTER_MAX_TOKENS=4000` (`config.py:71`).

**Вывод для доработки:** чтобы заявленное в брифе деление «по токенам/абзацам»
действительно работало, нужно заменить обрезку на токен-ориентированный чанкинг
с несколькими запросами и слиянием результатов (см. пункт 1.1 в `CODE_REVIEW.md`).

## 4. Как формируется и отправляется запрос

### 4.1 Gemini (`gemini_provider.py`)

1. `build_prompt_payload(...)` / `build_document_review_payload(...)` →
   сериализуется в JSON (`model_dump_json(ensure_ascii=False, indent=2)`).
2. JSON подставляется в шаблон промпта через `.format(...)`
   (`{comparison_payload}` / `{document_payload}`).
3. Вызывается `self._gateway.generate_json_result(prompt, ChangeReport,
   max_output_tokens=4000)` — гейтвей сам валидирует ответ под схему
   `ChangeReport`.
4. Ответу проставляется `provider="gemini"`, `model=<имя модели>`.
5. `clean_report(...)` выкидывает из ответа все `source_change_ids`, которых нет
   в исходном payload (защита от галлюцинаций модели).

Параметры Gemini берутся из `config.py`: модель `GEMINI_MODEL`,
`GEMINI_TEMPERATURE=0.1`, лимиты RPM/TPM/RPD, таймаут, ретраи.

### 4.2 OpenRouter (`openrouter_provider.py`)

То же построение payload и промпта, но запрос идёт напрямую по REST:

```jsonc
POST {OPENROUTER_BASE_URL}/chat/completions
{
  "model": "google/gemma-4-31b-it:free",
  "messages": [{ "role": "user", "content": "<промпт с JSON внутри>" }],
  "temperature": 0.1,
  "max_tokens": 4000,
  "response_format": {
    "type": "json_schema",
    "json_schema": { "name": "change_report", "strict": true,
                     "schema": <JSON-schema модели ChangeReport> }
  }
}
```

Ответ парсится из `choices[0].message.content`. Поскольку некоторые модели всё
равно оборачивают JSON в ```` ```json ````, есть функция `_extract_json_content`,
которая снимает обёртку и вырезает объект между первой `{` и последней `}`.
Затем — валидация в `ChangeReport` и тот же `clean_report`.

Весь промпт передаётся **одним сообщением с ролью `user`** — отдельного
системного сообщения нет, инструкция и данные склеены в один текст.

## 5. Промпты

Промпты лежат в `src/config.py`. Оба требуют ответ строго по JSON-схеме
`ChangeReport` и **на русском языке**. Общий блок `_RISK_DEFINITION` задаёт, что
считать финансовым риском.

### 5.1 Определение финансового риска (`_RISK_DEFINITION`)

Ключевая идея: **риск ≠ цена**. Согласованная цена, количество, налоги и обычный
график платежей — это обычные коммерческие условия, НЕ риск. Риском считается
условный/асимметричный денежный «навес»:

- штрафы, пени, неустойка, liquidated damages;
- последствия просрочки поставки/оплаты;
- ответственность, особенно неограниченная, убытки;
- индексация / одностороннее изменение цены;
- невозвратный аванс, удержание, потеря обеспечения/депозита.

`risk_type` обязан быть одним из:
`penalty | liability | indexation | price_change | prepayment | withholding | other`.

### 5.2 Промпт сравнения (`COMPARISON_ANALYSIS_PROMPT`)

Суть инструкции модели:

- анализировать **только** переданный diff, не выдумывать пункты;
- вернуть единый список `changes`, по каждому изменению:
  - `description` — одно самодостаточное предложение для юриста;
  - `source_change_ids` — id из payload, которые покрывает пункт (только
    существующие id);
  - `financial_risk` — `true` только для настоящего риска по определению выше;
  - `risk_type` — тег из списка, если риск, иначе `null`;
  - `estimated_impact` — денежный эффект (значение или короткая формула из сумм,
    которые есть в документе); `null`, если посчитать нельзя; выдумывать суммы
    запрещено;
- дополнительно: `summary` (1–2 предложения), `overall_risk_level`
  (`low|medium|high`), `recommended_review_points` (что проверить юристу).

В конце шаблона подставляется `Comparison payload: {comparison_payload}`.

### 5.3 Промпт аудита документа (`DOCUMENT_ANALYSIS_PROMPT`)

Похож, но цель — аудит одного договора на финансовый риск:

- анализировать только переданные блоки, ссылаться по `block_id`;
- приоритет — именно финансовые риски; можно добавить несколько ключевых
  коммерческих условий (цена, график оплаты) как контекст с
  `financial_risk=false`, но согласованная цена — это контекст, а не риск;
- `estimated_impact` по возможности количественно оценивает экспозицию, используя
  суммы из самого договора (например, общую цену);
- те же `summary`, `overall_risk_level`, `recommended_review_points`.

В конце подставляется `Document payload: {document_payload}`.

## 6. Формат ответа модели (`ChangeReport`)

```jsonc
{
  "summary": "Краткое резюме на 1-2 предложения",
  "overall_risk_level": "low | medium | high",
  "changes": [
    {
      "description": "Что изменилось/в чём риск — одно предложение",
      "source_change_ids": ["c12", "c13"],   // или block_id: ["p-0"]
      "financial_risk": true,
      "risk_type": "penalty",                  // null, если не риск
      "estimated_impact": "10% от суммы поставки" // null, если не оценить
    }
  ],
  "recommended_review_points": ["...", "..."],
  "provider": "gemini",    // проставляется кодом, не моделью
  "model": "gemini-..."    // проставляется кодом
}
```

После получения ответа:

- `clean_report(...)` удаляет несуществующие `source_change_ids`;
- `provider`/`model` перезаписываются кодом (нельзя доверять модели в этом);
- если был использован fallback после ошибки LLM, `provider` =
  `fallback_after_llm_error`.

> Примечание: `overall_risk_level` и `risk_type` сейчас — свободные строки
> (`str`), а не enum, поэтому формально модель может вернуть значение вне списка.
> Это отмечено как доработка в `CODE_REVIEW.md` (п. 2.4).

---

## 7. Детерминированный fallback (без LLM)

`FallbackInsightProvider` повторяет ту же контрактную форму ответа, но без
модели:

- ищет в тексте ключевые слова риска (`_RISK_KEYWORDS`: штраф, пеня, неустойка,
  ответственность, убытки, просрочка, удержание, индексация, penalty, liability
  и т. д.);
- вытаскивает суммы/проценты регуляркой `_MONEY_OR_PERCENT_RE`;
- классифицирует `risk_type` по найденным словам;
- для аудита берёт до 20 «рисковых» блоков (или первые 5, если рисков не нашёл).

Это гарантирует, что демо работает даже без ключей API, но качество ниже, чем у
LLM.

---

## 8. Краткие итоги

- В модель уходит **структурированный JSON** (diff или блоки), а не сырой текст
  целиком.
- Запрос — **один на документ**, ответ строго по JSON-схеме `ChangeReport`, на
  русском.
- Настоящего деления под контекстное окно **нет**: применяется обрезка
  (60 изменений / 120 блоков / 1400 символов) — это главный пункт к доработке.
- Промпты живут в `src/config.py`, центральное понятие — отличать финансовый
  **риск** от обычной согласованной **цены**.
- Ответ модели считается недоверенным: id-ссылки чистятся, `provider`/`model`
  проставляются кодом, при сбое LLM включается детерминированный fallback.
